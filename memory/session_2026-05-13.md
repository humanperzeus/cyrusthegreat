## Decisions / shipped / verified

### Read-path resilience (independent of wallet RPC health)

1. **Multi-URL fallback per chain**: `NETWORKS[cid].rpcUrls` expanded from 1 URL to 3-5 per chain (Sepolia now has 5 candidates: rpc.sepolia.org, two publicnode variants, blastapi, 1rpc).

2. **Probe-on-connect** (`probeReadRpc(cid)`): pings each URL with `eth_chainId` at connect time; picks the first that returns the correct chainId. Logs every probe with green/red status. Reveals exactly which public RPCs the browser can/can't reach (Brave Shields blocks some — that surfaced in the log as `Failed to fetch` on rpc.sepolia.org).

3. **FallbackProvider wrapper**: `getReadProvider()` now returns an `ethers.FallbackProvider` configured with `quorum: 1` across all the chain's rpcUrls. If one URL rate-limits mid-burst (common in Brave Shields under Test 4's 6+ parallel RPC calls), ethers silently rotates to the next URL. No user-visible retry needed.

4. **Optional Alchemy override**: if the user pastes their Alchemy URL into the existing "API health check" input, it's prepended to the candidate list AND persisted to localStorage. Reads switch to Alchemy on next probe. Not required for ops — public RPCs work with FallbackProvider — but available for users behind aggressive ad-blockers.

5. **Header pill** (`RPC: <url> (+N fallbacks)`): always-visible indicator of which RPC is currently primary, so the read-path can never be silently broken.

### Error surfacing (`formatErr` helper)

Ethers v6 wraps multiple failures in opaque `could not coalesce error`. New `formatErr()` walks the entire `e.cause` / `e.error` / `e.errors` / `e.info` chain, printing each level with indent + method + code + data. Applied to:
- All Test card catch blocks (was `e.shortMessage` only)
- `refreshStateFor`'s per-call error panel (was generic message)

This turned out to be **the** unblock: once Test 3's catch showed `• Request failed with status code 404 / code: -32000` at the second level, the wallet-RPC theory became testable.

### Wallet RPC diagnostic (🔧 Diagnose wallet button)

New header button that fires 9 JSON-RPC methods directly at `window.ethereum.request()` and renders ✓/✗ + response into a visible panel under the header. Methods tested:
- Locally-answerable: `eth_chainId`, `eth_accounts`, `net_version`
- Chain-touching: `eth_blockNumber`, `eth_gasPrice`, `eth_getBalance`, `eth_getTransactionCount`, `eth_feeHistory`, `eth_maxPriorityFeePerGas`

Pattern revealed: on Zerion, all 3 locally-answerable methods green; all 6 chain-touching methods 404. Diagnostic isolated the failure to **the wallet's RPC backend**, independent of any dapp code.

### Pre-populated transaction helper (`prepareTx`)

Builds a fully-populated overrides object (nonce, chainId, maxFeePerGas, maxPriorityFeePerGas, gasLimit) from the read provider. Means writes can in principle bypass the wallet's broken-RPC pre-flight, hitting it only for the final `eth_sendTransaction`. Currently applied to Test 3 only; ready to roll through Tests 4–12 if a future wallet's broadcast endpoint is healthy but its pre-flight reads aren't.

### Sign-and-broadcast fallback — abandoned (didn't work)

Tried `sendViaSignAndBroadcast()`: call `eth_signTransaction` to get raw bytes, then broadcast via working public RPC. Result: Rabby (and most modern wallets) don't implement `eth_signTransaction` — they proxy it to the user's configured RPC. Public RPCs don't support `eth_signTransaction` (no privkey held), so the call 404s. **No path exists to bypass the wallet's broadcast** from dapp JS. Removed the fallback after confirming it caused false-positive matches on Rabby.

## Lessons learned this session (added to tech_learnings.md)

- **L-010**: Wallet RPC health is a write-path-only failure mode; reads can be insulated via dapp-controlled providers, writes can't. Diagnose with a per-method probe of `window.ethereum.request()` before debugging dapp code.

## Phase B-2: CyrusTresor1 design (completed this session)

Designed end-to-end via guided question rounds. Decisions baked into `docs/cyrustresor1_spec.md` (247 lines).

Core choices:
- Pattern A: Bank8 preserved byte-for-byte + opt-in anonymity pool layer
- 1-hour epochs, 1-epoch minimum reveal wait
- Buckets configured per chain at deploy time (constructor params); testnet defaults 0.001/0.01/0.1/1 ETH
- Commit-only fees, free reveals (enables teleport UX where sender pre-pays, recipient claims free)
- Wallet-only pool entry (no vault-to-pool, prevents linkability)
- Front-running protection: `withdrawTo` baked into commitment hash
- ZK verifier slot reserved, constructor-immutable (no admin keys; v2 = new contract)
- All pool storage keyed by commitment hash, never user address (avoids Portal11's global-state-clearing bug class)
- Teleport = dapp UX layer, not a contract function
- Threat model: chain-analyst MVP, state-level extensible via v2 ZK redeploy
- Contract name: `CyrusTresor1.sol`

Spec is implementation-ready. Suggested next-session workflow detailed in the spec's "Implementation handoff" section.
