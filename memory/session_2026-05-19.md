## Decisions / shipped / verified

### v1 chain switcher fix (commit `148ed16`)

Root cause: `src/metamask.js` helpers invoke `window.ethereum.request` directly to issue `wallet_switchEthereumChain`. The dapp uses Reown AppKit's WalletConnect connector — under WalletConnect, the wallet talks through a relay, NOT through `window.ethereum`. So the request never reaches Rabby's actual session.

Fix: replace `switchToChain()` in `src/config/web3.ts` to use wagmi's imperative `switchChain(config, { chainId })`. Routes through the active connector — works for injected wallets AND WalletConnect.

Verified by user post-deploy: tested tBSC chain switch on v1 → Rabby popup appeared → deposit + withdraw 0.1 tBSC succeeded.

### v2 ChainSwitcher (commit `adde46b`)

v2 had no chain-switcher UI at all — pool users couldn't change chains without bouncing back to v1. Added `src/components/pool/ChainSwitcher.tsx` (~95 LOC) mirroring v1's button style. Detects wallet-vs-dapp chain mismatch and shows a "wallet on X, click Y to sync" warning.

### Notebook dedup + /claim indexing grace + post-Approve allowance polling

Three small UX fixes from the USD1 test (commits `8e64d6c`, `17cfdcc`):

1. **Notebook dedup** — `revealFromURL` was adding a duplicate entry every time, even when the original entry was already in the notebook. Same commitment hash, two entries. Fixed at hook level (check before insert) AND display layer (merge duplicates).

2. **/claim "indexing" state** — distinguish "freshly committed, RPC hasn't indexed yet" from "truly unknown commitment". 45s grace period during which the page shows a yellow "Checking on-chain state…" with a Refresh button, then escalates to the existing red "unknown commitment" error.

3. **Approve→Commit transition** — `approveToken` waits for the receipt then polls `allowance()` directly via publicClient for up to 8s (4 attempts × 2s) until the new value is observed. Previously relied on a single `refetchAllowance()` which could miss the cache update.

### Decimal-aware display (commit `12c8361`)

Notebook + Claim were using `formatEther` everywhere. Sepolia hides the bug (USD1 = 18 decimals), but mainnet USDC = 6 decimals: a 10 USDC bucket would render as "0.00000000001 USDC". Fix:

- `NotebookEntry` interface: 3 new optional fields stored at commit time — `bucketSizeWei`, `tokenDecimals`, `tokenSymbol`. Older entries fall back to "bucket N" label.
- `commit()`: populates the new fields from POOL_TOKENS_BY_CHAIN registry.
- Notebook entries display "10 USD1" instead of "bucket 0 (USD1)".
- Claim page reads `usePoolBucketSizes(token)` + `useTokenDecimals(token)` and shows "You can claim 10 USD1 to 0x…" in the header.
